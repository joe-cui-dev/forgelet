# Web Source Text Enters Active Context as Appended Conversation Material

Three existing boundaries constrain where a mid-Session Web Source's complete text can live: the prompt's stable prefix must stay byte-identical across turns (ADR 0032, enforced by a prefix-stability test), the Trace must not contain complete source bodies (ADR 0038), and a `web_read` tool observation is only a bounded receipt (ADR 0038). Rendering dynamically acquired sources into the launch-time attachment block would grow the stable prefix mid-Session, forking the provider prompt cache on every read and failing the prefix-stability invariant.

We decided that after a successful `web_read`, the kernel appends one synthetic user-role message per new Web Source to the conversation — in tool-call order, after that batch's tool observations — carrying the source's ledger identity (id, title, final URL, content hash, byte count) and its extracted text bounded by the per-source injection limit. The message is foldable under the same discipline as Observation Digests: a digest keeps the source id, hash, and a bounded excerpt, because the complete text stays recoverable from the Session Source Ledger's persisted body. A deduplicated read injects nothing; its receipt names the existing ledger entry.

We rejected growing the stable attachment prefix (breaks ADR 0032 and re-bills the entire prefix each read), carrying the full text inside the tool observation (observations are the Trace's bounded evidence, and ADR 0038 keeps complete bodies out of the Trace), and exempting the injected message from folding (tail content follows fold discipline; exemption from folding is the stable prefix's privilege, and evicted text remains recoverable from the ledger).
