# Browser Workbench Streams Over Long-Lived Native Messaging

Browser-launched Sessions need an immediate protocol acknowledgement, a Session-ready identity, live progress, and one explicit terminal result, which one-shot `sendNativeMessage()` cannot carry. Browser Workbench invocations will therefore use a long-lived `connectNative()` port and an ordered multi-frame protocol with idempotent action/invocation identities, monotonic sequence numbers, typed pre-Session rejection, `sessionId` and Trace path after successful Session creation, wrapped Session Live Events, and exactly one completed, stopped, or failed terminal frame. The extension service worker owns the Native Port until that terminal frame; closing and reopening the Side Panel only detaches and reattaches presentation by invocation identity and never cancels the Session. Only an explicit, idempotent cancel frame means user cancellation: before Session creation it rejects without a Trace, while afterward it propagates cancellation through active model and Web requests and finishes the same Session as `stopped` with reason `user_stopped`; transport loss remains a failure rather than implied consent to stop. The existing `current-page.json`, `forge browser read-current`, and `--with-browser` path remains a compatibility Adapter during migration; this decision does not introduce a localhost daemon.
