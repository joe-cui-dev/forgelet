# Writing Project Sessions Gain read_workspace Scoped to Member Artifacts

The Writing Workflow has never held `read_workspace` — its grant is `read_context`, `update_plan`, `model_generate_text`, so a Writing Session sees only what is fed to it. Writing Project continuity breaks that assumption: continuing chapter 11 needs the model to consult earlier chapters on demand. Feeding every member artifact up front was rejected because a growing long-form work collides directly with the conversation budget and Rolling Summary machinery (ADR 0019); a persisted project synopsis was rejected because it is a new durable, model-written artifact that would drag in the whole user-approval and update-timing question, which is out of scope for a minimal slice.

Instead, when a Writing Project is active, the Session's Workflow Capability Grant additionally includes `read_workspace`, and the Session Read Scope is narrowed to exactly the project's member artifact paths. The Session receives the Continuation Head as its continuation attachment plus the member listing, and reads earlier members through the normal tool path — so every consultation of a past chapter is a real Trace event, the grant remains necessary but not sufficient (Permission Policy still applies, per ADR 0006), and a project Session still cannot roam the repository. Writing Sessions without a project keep the original grant unchanged.
