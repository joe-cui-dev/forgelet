# Prompt layout keeps a stable prefix and a volatile tail

Model providers with automatic prefix caching (DeepSeek, OpenAI-compatible APIs) only reuse a request's leading byte-identical prefix. Dogfood session `sess_mrc1hsmn` showed every turn hitting exactly 1024 cached tokens because the per-turn budget line sat near the head of the second message, forking the prefix immediately after the system prompt while the growing conversation (up to ~28K tokens per turn) missed entirely.

We decided the kernel's model-facing message layout is: constant system prompt, then stable task context (task, attachments, durable memory), then Rolling Summary, then the append-only conversation, then a single Turn Status message carrying everything that changes per turn (plan progress, budget consumption, compaction status, final-tool-turn and final-answer-only notices). The system prompt no longer varies with `finalOnly`; those instructions move into Turn Status, because the API-level empty tool list is the hard guard and the tail is the last thing the model reads.

One layout serves all providers — we rejected per-provider message layouts (two prompt behaviors to debug and test) and rejected merely coarsening the budget line (plan updates and wrapup notices would still fork the prefix). The wrapup/final-answer turn stays a full cache miss by design: it passes an empty tool list (ADR 0029's hard guard), which itself reforks the rendered prefix, so softening it for cache would trade a real safety boundary for at most one turn per Session. Observation Digest compaction still rewrites old messages in place when the conversation exceeds its byte budget (ADR 0013); its cache cost is accepted for now and evaluated separately. A prefix-stability test enforces the invariant: each turn's messages, minus the Turn Status tail, must be a byte-level prefix of the next turn's.
