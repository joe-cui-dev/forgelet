# Page Answer Evidence Must Match the Browser Capture

A Page Answer may carry up to three bounded Evidence excerpts, and each excerpt must match text in the immutable browser capture after deterministic whitespace normalization. Model-written paraphrases, page URLs, and heading names alone are not Evidence. If no supporting passage can be verified, the model must put the exact sentinel `No supporting passage in the captured page.` in Evidence; the normalizer converts it to `groundingStatus: "not_found"` with an empty evidence array. The sentinel is never rendered directly: the Side Panel displays a deterministic not-found message in the turn's selected output language and scopes the claim to captured content, with an additional partial-coverage warning when capture metadata says content was truncated. This is a successful Page Answer, advances the Page Conversation Head, and enters Page Conversation History. The normalized Answer, grounding status, and verified excerpts are user-visible final output and therefore enter the child Session's `final_summary` Trace evidence and disposable browser projection; the full capture body remains only in its workspace capture file. A merely empty Evidence section, missing required sections, excess excerpts, or any non-matching excerpt fails the child Session with typed reason `invalid_page_answer`; the normalizer does not infer intent, silently discard Evidence, or make a second model repair call.
