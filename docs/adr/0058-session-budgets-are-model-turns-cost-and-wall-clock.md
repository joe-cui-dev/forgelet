# Session Budgets Are Model Turns, Estimated Cost, and Wall Clock

The kernel's founding commit introduced a cumulative input-token budget (`maxInputTokens`) alongside the model-turn and estimated-cost limits, and the three were never calibrated against one another. Because every model turn resends the whole conversation, cumulative input grows roughly quadratically as a Session accumulates material, so a Session running near the Active Context byte cap could afford only 3–4 of its 12 permitted turns before the token budget bound. Real web-learning Sessions hit this wall while estimated cost sat under 1% of its limit: the token budget re-charged what the cost budget already prices correctly (including provider cache economics) and double-guarded the single-turn context size that compaction already bounds. It guarded no distinct resource of its own.

We decided to remove the cumulative input-token budget. Session Budgets are exactly three dimensions, each guarding one distinct resource: model turns guard progress, estimated cost guards money, and wall clock guards the user's time. Token usage stays recorded in usage telemetry and `budget_update` Trace events — recording usage and limiting it are decoupled — but tokens no longer stop a Session.

Removal makes estimated cost the only usage-proportional guard, so unknown cost may no longer be silently counted as zero (previously a model missing from the pricing table accumulated $0 and the cost budget could never bind). A turn whose usage is missing or whose model has no pricing increments `unpricedTurns` in usage, marking the Session's estimated cost as a lower bound. The Session keeps running — a genuinely free model is a legitimate configuration, and model turns plus wall clock structurally bound the worst-case exposure — but the degradation is visible in Turn Status, the Trace, and a best-effort launch notice, never hidden.

Any budget dimension that can stop a Session must be visible to the model in Turn Status; wall clock joins turns and cost there. This invariant binds future budget dimensions too — it exists so no Session again dies against a wall the model could not see and plan around.

We rejected raising the limit or overriding it per workflow (tuning a budget that guards no distinct resource), re-scoping it to per-turn peak input (a second guard, in different units, on the single-turn context that `maxConversationBytes` and compaction already bound), and billing unpriced usage at a synthetic conservative rate (fabricated cost is as dishonest as silent zero). Folding Web Source injections earlier remains a worthwhile efficiency optimization but is deferred: with the cumulative budget gone, resent bytes are priced correctly by the cost budget instead of hitting a false wall. Removal is complete rather than staged: no deprecation period, old Traces keep their recorded `maxInputTokens` as honest history, stale fields in Pause Snapshots and config are tolerated on read, and a config that still sets `budgets.maxInputTokens` earns a one-time notice that it is no longer enforced.
